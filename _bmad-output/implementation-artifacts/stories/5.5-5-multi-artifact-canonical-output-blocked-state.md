# Story 5.5.5: Multi-Artifact + Canonical-JSON Output + `blocked` Job State

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a consultant,
I want a single run to persist multiple artifacts (canonical JSON + Excel), expose the typed canonical output, and surface a `blocked` job state when a QA egregious flag fires,
So that the full structured result is captured and downstream skills refuse to consume an unsafe output.

## Acceptance Criteria

**AC1 — Multi-artifact persistence via the established `output_files` shape (FR-OUT-01, FR-OUT-05):**
Given a code-driven hybrid returns N artifacts (its `envelope.artifacts`) + a `canonical` typed output
When the job completes
Then the executor uploads every artifact file (from the skill's `output_dir`) to `StorageProvider` and records them in `result_metadata["output_files"]` — the **existing Story 3.6 shape** `[{key, format, content_type, size_bytes}]` — so the job READ API presigns each (already implemented) and the Run Console renders each as an individual download (already implemented). The canonical JSON remains independently retrievable as its own artifact entry (`format: "json"`, content-type `application/json`) AND `result_metadata["canonical"] = {schema_ref, schema_version, data}` carries the typed canonical for downstream binding. **No migration** — `output_files` is JSONB inside `result_metadata` (LOCKED DECISION: reuse the 3.6 mechanism, do not add an InvocationResult column).

**AC2 — New terminal job status `blocked`, distinct from `failed` (FR-EXE-09):**
Given the skill's envelope reports `status: "blocked"` (an egregious QA flag fired)
When the job finishes
Then the job reaches a new terminal status `blocked` — `JOB_STATUS_BLOCKED = "blocked"` added to the status constants and to `JOB_TERMINAL_STATUSES` (so `_guard_not_terminal` treats it as terminal — a `blocked` job is never re-transitioned) — **distinct from `failed`** (the run succeeded; the OUTPUT is held for human review). The artifacts + canonical from AC1 are STILL persisted on a `blocked` job (the consultant needs to see what was produced). No DB migration: `InvocationJob.status` is `VARCHAR(16)` with no DB enum/CHECK (verified) — `blocked` (7 chars) fits.

**AC3 — Run Console surfaces `blocked` as human-resolution-required (FR-EXE-09):**
Given a job in `blocked` status
When viewed in the Run Console
Then it shows a distinct `blocked` badge (amber/caution tone, NOT the red `failed` tone), labeled to convey "output held — needs human review", and polling stops (the FE `TERMINAL_STATUSES` set includes `blocked`). The persisted artifacts are still individually downloadable (the consultant reviews them to resolve the block).

**AC4 — Multi-artifact downloadable via presigned URLs:**
Given the multi-artifact result (completed OR blocked)
When displayed in the Run Console
Then all artifacts are individually downloadable via presigned URLs (24h TTL) — this path already exists for `output_files` (RunConsole.tsx:908-962 + jobs READ API); AC4 is satisfied by AC1 populating `output_files` correctly, plus a regression test proving a multi-artifact code-driven result renders N download links.

**AC5 (DEFERRED — explicitly OUT of scope, recorded for traceability):** The epic's "a downstream skill refuses to consume a `blocked` upstream output" rule is **deferred to the future pipeline epic** (LOCKED DECISION). There is no pipeline-consumption mechanism today (no `upstream_job_id` on `InvocationRequest`, no upstream-job-consumption seam), so the refusal has no consumer to guard. 5.5.5 ships the producer side (the `blocked` state) only. See Dev Notes "Deferred: consumes-refusal" and the deferred-work.md entry.

## Dev Notes

### Where 5.5.5 sits — and what's already built (read this first)

5.5.5 is the **output half** of the code-driven hybrid path: 5.5.3 built the executor (returns canonical only), 5.5.4 fed it the raw file. 5.5.5 makes a run persist **all** its artifacts and surfaces the **`blocked`** QA gate. It is **full-stack** (LOCKED) — but the heavy lifting is backend; the FE is small because **the multi-artifact download UI already exists** from Story 3.6.

**Two things are already built — DO NOT rebuild them:**
1. **`result_metadata["output_files"]` multi-artifact mechanism (Story 3.6).** The jobs READ API (`app/api/v1/jobs.py:132-178`) already loops `result_metadata["output_files"]`, presigns each (`OutputFileRef{key,format,content_type,size_bytes,url}`), and returns them in `JobResult.output_files`. The Run Console (`RunConsole.tsx:908-962`) already renders each as a download. **5.5.5's job is to POPULATE `output_files` from the code-driven executor** — not to invent persistence. (LOCKED DECISION: reuse this, no InvocationResult migration.)
2. **The envelope already carries `status: "ok"|"blocked"`** (`CodeDrivenResultEnvelope`, `code_driven_executor.py:104`) — it's validated but currently IGNORED (the worker always `mark_completed`s). 5.5.5 wires it through.

### VERIFIED real-skill envelope shape (the reshape this story owns)

From the real `velara-protocol-extractor` (`~/Downloads/velara-protocol-extractor/src/velara_extractor/plugin.py`, read during 5.5.4 creation — see [[project-client-skill-contract]] / the epic's client-skill section). `run()` returns:
```python
{
  "status": "ok" | "blocked",                 # blocked = an egregious QA flag fired
  "schema_version": ext.metadata.schema_version,   # REAL value "0.3.0" (model.py:33), NOT manifest's stale 0.1.0
  "canonical": {... ProtocolExtraction as dict ...},
  "billing_grid": [ ... ],                    # extra top-level key
  "artifacts": {"canonical_json": path, "excel": path|None},   # ← a DICT, role→path
  "qa": {"releasable": bool, "egregious": [...], "warnings": [...]},
}
```
**This is the contract 5.5.5 must accept.** The current `CodeDrivenResultEnvelope.artifacts: list[dict]` is WRONG for the real skill — it raises `CODE_DRIVEN_ENVELOPE_ERROR`. Task 1 changes it to `artifacts: dict[str, str | None]`. `status: "blocked"` fires when `not is_releasable(ext)` (an egregious QA flag) — exactly the case AC2 surfaces. This was logged as a `[BLOCKS 5.5.6]` item in deferred-work.md under the 5.5-4 entry; 5.5.5 closes it.

### The executor end-state today (the exact code Task 1 replaces)

`code_driven_executor.py:441-479` persists ONLY canonical.json and returns `(canonical_key, result_metadata)` with `result_metadata` carrying `artifact_count` (a count, no actual file persistence) and `qa`. The `envelope.artifacts` files sit in `output_dir` and are **dropped on cleanup** (the `finally: shutil.rmtree(workspace)` at line 481-484 deletes them). Task 1 must read+upload them BEFORE the workspace is torn down — the upload loop runs inside the `try`, before `finally`. (The skill writes artifacts to `output_dir`, which is `workspace/output_dir` — already created at line ~177.)

### `blocked` status — the model + guard mechanics

- `JOB_TERMINAL_STATUSES` (`invocation.py:39`) is used by `_guard_not_terminal` (`job_service.py:310-327`) to make terminal transitions idempotent (a Celery re-delivery must not flip or double-write). Adding `blocked` there means a `blocked` job is correctly frozen.
- A `blocked` job HAS an `InvocationResult` (artifacts persisted) and `completed_at` set, but `error_code = NULL`. This is the structural difference from `failed` (which sets `error_code`, no result row). The Run Console reads the QA reasons from `result.result_metadata.qa.egregious`.
- `mark_blocked` mirrors `mark_completed` (NOT `mark_failed`) — it writes the result row. Copy `mark_completed` (`job_service.py:344-369`) and change only the status constant.

### Deferred: consumes-refusal (AC5 — explicitly out of scope, LOCKED)

The epic AC "a downstream skill attempting to consume a blocked upstream output is refused before it executes" presupposes a **pipeline-consumption mechanism that does not exist**: `InvocationRequest` (`schemas/invocation.py:17-50`, `extra="forbid"`) has `file_ref_ids` but NO `upstream_job_id`; there is no `assert_upstream_job_consumable` seam (only `assert_file_ref_ready` for uploaded docs). Building it now = a net-new pipeline feature with no consumer until the downstream coding/coverage skills exist (a future epic). **LOCKED DECISION: defer.** 5.5.5 ships the producer side (the `blocked` state). Log a deferred-work entry: "consumes-refusal of a blocked upstream — needs `upstream_job_id` on InvocationRequest + `assert_upstream_job_consumable` (refuse if upstream `blocked`/`failed` + match downstream `consumes.schema_ref` against the upstream producer) — deferred to the pipeline epic; the `consumes` declaration (5.5.2) + the `blocked` state (this story) are the prerequisites, now both in place." The 5.5.2 `consumes` column (`app/models/skill.py:103-108`) is the field that refusal will eventually check against — it already exists.
