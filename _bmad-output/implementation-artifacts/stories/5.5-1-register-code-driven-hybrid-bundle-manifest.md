# Story 5.5.1: Register a Code-Driven Hybrid Skill (Bundle Artifact + Manifest)

Status: done

## Story

As an MA Tech developer,
I want to register a hybrid skill whose artifact is a multi-file bundle with a declared entrypoint, dependency lockfile, schemas, secrets, and egress declarations,
So that code-driven hybrid skills can live in the registry as immutable versioned artifacts alongside prompt/code/LLM-driven-hybrid skills.

## Acceptance Criteria

**AC1 — Bundle stored as immutable versioned artifact:**
Given a hybrid skill bundle (zip/dir) with a manifest declaring `entrypoint` (`module:callable`), `requirements`/lockfile, `input_schema`, `output_schema` + `schema_version`, `requires` (secrets), and declared egress destinations
When I register it via the skill API
Then the **bundle** is stored as one immutable versioned artifact via `StorageProvider` with `runtime_type = "hybrid"` (FR-REG-01), and the manifest metadata is persisted and retrievable (FR-REG-03)

**AC2 — Missing required manifest field → 422:**
Given a bundle missing a required manifest field (entrypoint, `output_schema`/`schema_version`, or lockfile)
When I attempt to register it
Then registration is rejected 422 with a clear, specific error naming the missing field (FR-REG-03)

**AC3 — LLM-driven hybrid is unaffected:**
Given the existing LLM-driven hybrid manifest shape (`{system, tools[], code}`)
When it is registered
Then it still validates and is unaffected — manifest-shape detection distinguishes the two without a new `runtime_type` (both hybrid shapes coexist)

## Dev Notes

### Architecture Context

This story **extends** the skill registry (Epic 2, closed) via additive schema/service changes. It does NOT reopen Epic 2; it adds the code-driven hybrid registration path that Epic 2 was always designed to support (FR-REG-03 "input schema, output schema" was designed-in but unused).

**The fundamental design:** A code-driven hybrid skill's artifact (`initial_content`) is a JSON string (the manifest) uploaded to S3 as the immutable versioned blob — exactly like the LLM-driven hybrid's `{system, tools, code}` JSON. The distinction is manifest shape, not `runtime_type`. Both shapes share `runtime_type = "hybrid"`. Shape is detected at registration time (Task 2) and at execution time (Story 5.5.3).

### Critical Pattern: `create_skill` artifact validation hook (Task 2)

Read `skill_service.py:214-311` carefully before touching it. The current flow:
1. Validate name/description
2. Write artifact to S3 (`_write_artifact`)
3. Flush skill row (current_version_id=NULL)
4. Flush SkillVersion row
5. Update skill.current_version_id + commit
6. On error: rollback + `_safe_delete(artifact_key)`

**Insert the validation at step 2.5** (after S3 write, before first flush). Reason: parse `initial_content.encode()` in memory (the S3 write uses the same bytes). If validation fails, call `_safe_delete(artifact_key)` and re-raise — same cleanup discipline as the existing error path.

```python
# pseudo-code — insert in create_skill after artifact S3 write
if runtime_type == "hybrid":
    raw = initial_content.encode()
    if is_code_driven_manifest(raw):
        manifest = parse_code_driven_manifest(raw)  # raises InvalidCodeDrivenManifestError on bad shape
    else:
        parse_hybrid_artifact(raw)  # raises InvalidHybridArtifactError on bad LLM-driven shape
```

### Critical Pattern: `schema_version` and `egress` population

Only populate from the manifest. In `create_skill`, after the code-driven manifest parse:
```python
if is_code_driven:
    skill.schema_version = manifest.schema_version
    skill.egress = manifest.egress  # list[str], default []
```
For LLM-driven and other runtimes: leave both at their DB defaults (NULL / `[]`).

In `create_version()` (lines 438-541), same pattern: parse the new artifact, update `skill.schema_version` and `skill.egress` in the **same transaction** as setting `skill.current_version_id`. This ensures schema_version always reflects the current version's declaration.

### Existing seams to reuse

**`requires` validation pattern** (`app/schemas/skill.py:44-84`): `_normalize_requires()` validates token format, de-dupes, enforces bounds. **Do NOT replicate this for egress** — egress is free-form strings (hostnames/URLs), not structured tokens. Parse and validate egress only inside `CodeDrivenHybridManifest` (a list of non-empty strings is sufficient for Phase 1; enforcement is advisory at Phase 1 per the open question in the epic).

**`_safe_delete`** (skill_service.py): already handles StorageProvider errors gracefully — reuse it on manifest validation failure.

**`InvalidHybridArtifactError`** (`hybrid_artifact.py`): the template for `InvalidCodeDrivenManifestError`. Mirror the same pattern: `VelaraHTTPException` subclass, 422, stable code `INVALID_CODE_DRIVEN_MANIFEST`, no artifact content in the message.

**Semver validation**: `_parse_semver()` and `_SEMVER_RE` at `skill_service.py:155-183` — import and reuse in `code_driven_hybrid.py` for `schema_version` validation. Do NOT define a second semver regex.

### Code-driven manifest JSON shape (canonical reference)

```json
{
  "entrypoint": "velara_extractor.plugin:run",
  "requirements": "pdfplumber==0.11.4\nPyMuPDF==1.24.9\nopenpyxl==3.1.5\nanthropic==0.50.0\npydantic==2.7.4\n",
  "output_schema": {"type": "object", "properties": {"protocols": {"type": "array"}}, "required": ["protocols"]},
  "schema_version": "0.3.0",
  "input_schema": {"type": "object", "properties": {"file_ref_ids": {"type": "array"}}},
  "egress": ["api.anthropic.com"],
  "description": "Velara Protocol Extractor — extracts protocol elements from clinical trial PDFs"
}
```

**vs. LLM-driven manifest (Story 3.5):**
```json
{
  "system": "You are a helpful assistant...",
  "tools": [{"name": "fetch_data", "description": "...", "input_schema": {...}}],
  "code": "def fetch_data(url): ..."
}
```

Detection rule: code-driven has `entrypoint` key; LLM-driven has `system`/`tools`/`code` keys. Never overlap.

### PHI / IP discipline

The manifest content (entrypoint path, requirements text, output schema, egress list) is NOT PHI — it is metadata. However:
- Do NOT log the full `requirements` text (can be large; log count of packages instead)
- Do NOT log `output_schema` content (log presence only)
- `schema_version` and `entrypoint` may be logged (metadata)
- The artifact bytes (the full manifest JSON) travel only to S3 and are never returned by GET endpoints — same IP protection as prompt/code/LLM-hybrid artifacts (`SkillRead` deliberately omits `artifact_key`, line 3 of `schemas/skill.py`)
