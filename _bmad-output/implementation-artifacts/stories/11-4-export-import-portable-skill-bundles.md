# Story 11.4: Export / Import Portable Skill Bundles

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Vitalief operator,
I want to export a skill+version to a portable **signed** bundle and import it into another environment,
so that a skill built in dev can move to staging/prod without hand re-registration — and **trust never rides along** (the imported skill re-enters certification in the target environment).

## Acceptance Criteria

1. **AC1 — Export produces a signed, content-addressed portable bundle.**
   **Given** a skill version (single-object inline OR multi-file bundle)
   **When** an admin/ma_tech operator exports it
   **Then** the platform produces a **portable bundle** — a ZIP containing (a) the skill's artifact files byte-for-byte (the inline artifact as its stored object, or every `artifact_set` member), (b) the code-driven `manifest.json` when the version is a bundle, and (c) a top-level `velara-export.json` **envelope** carrying the skill+version metadata (name, description, author, `runtime_type`, `visibility`→exported as advisory only, `scope`, `output_format`, `tags`, `requires`, `location_dependent`, `input_schema`/`output_schema`, `consumes`, `schema_version`, `egress`, the source `version` string, source `org_id`, source `environment`, a per-file `sha256` manifest, and a **content-address digest** over the sorted `(path, sha256)` pairs) — plus an **HMAC-SHA256 signature** over the canonical envelope bytes, computed with a per-environment signing key from `Settings` (never in the bundle, never in code). The export is content-addressed (reuses the existing `_compute_bundle_record` sha256 discipline). The response is a presigned **download URL** to the assembled bundle object in the skill bucket (mirrors the existing `presign_download` GET-URL file-serving pattern — [jobs.py:145](../../../velara-api/app/api/v1/jobs.py#L145) / [client.py:328](../../../velara-api/app/api/v1/client.py#L328), NOT the `/bundle/presign` PUT route), NOT raw bytes through the API.

2. **AC2 — Import validates the bundle and re-creates the version immutably; a tampered/unsigned bundle is rejected.**
   **Given** an operator uploads a previously-exported bundle into a target environment (via the SAME presign → PUT → confirm flow Story 11.1 uses for ZIP upload)
   **When** import runs
   **Then** the platform (a) verifies the HMAC-SHA256 **signature** against the target environment's signing key via `hmac.compare_digest` — an **unsigned** bundle → `422 BUNDLE_SIGNATURE_MISSING`, a **bad signature** → `422 BUNDLE_SIGNATURE_INVALID`; (b) recomputes every per-file `sha256` and the content-address digest and rejects any **mismatch** (bytes tampered after signing, or an envelope that disagrees with the files) → `422 BUNDLE_TAMPERED`; (c) rejects a **structurally malformed** export (missing `velara-export.json`, unparseable envelope, wrong schema) → `422 INVALID_EXPORT_BUNDLE`; then (d) re-creates the skill + version through the **immutable** registry path (the same code-driven-hybrid bundle path for a bundle export; the inline single-object path for an inline export) — nothing is persisted on any rejection (validate before any S3 write / DB insert). The imported version preserves the **source version string** (e.g. `2.3.0`), not a reset to `1.0.0`.

3. **AC3 — Import lands the skill in a non-`client_ready` state so target-environment certification must run.**
   **Given** a successful import
   **When** the skill+version is created in the target environment
   **Then** it lands in `draft` (the create default) — **never** `client_ready` — with **no certification records copied** across environments; the imported skill's `org_id`/`created_by_user_id` are the **importing** caller's, and the two-key certification gate (`assert_certified_for_client_ready`) re-runs against the target org before the skill can advance to `client_ready`. This is the load-bearing rule: *trust does not copy across environments by file transfer.* The source's `visibility`/`lifecycle_state` are informational in the envelope but do **not** grant the imported skill any trust; `ai_adapted` (if present in the manifest) is preserved (it is part of the artifact bytes, not a trust flag).

4. **AC4 — Export/import are grantor-gated (admin / ma_tech), internal-only, and PHI/IP-clean.**
   **Given** the export and import endpoints
   **When** they are called
   **Then** both are `RejectClient`-gated (router-level, already in place) AND per-route `RejectNonGrantor`-gated (admin/ma_tech only — mirror the propose route). The exported bundle contains **ONLY the skill artifact** — code, manifest, schemas, metadata — and **NEVER** any invocation input/output, document reference, run history, or audit row (*if it isn't the skill itself, it isn't in the bundle*). Export reads only the registry `Skill`/`SkillVersion` + its `StorageProvider` artifact; it touches no `invocation_*` / `audit_*` / ingest table. Logs and error messages never echo artifact bytes, envelope field *values*, or S3 keys (IP discipline) — a rejection names the failure class (`BUNDLE_TAMPERED`, etc.), never the offending bytes.

5. **AC5 — Both export and import record a best-effort admin audit event.**
   **Given** a successful export or import
   **When** it completes
   **Then** it records a new admin audit event — `admin.skill_exported` on export, `admin.skill_imported` on import — via the EXISTING best-effort `record_admin_action` seam (`hierarchy_path="org"`, after the operation, never rolling back the operation on an audit-write failure; metadata = skill_id / source+target version / source environment / content-address digest — IDs and hashes only, never bytes). Two new `EVENT_ADMIN_SKILL_*` constants are added (none exist yet).

6. **AC6 — Frontend: an Export action on the skill detail and an Import affordance on the registry.**
   **Given** the internal skills surface
   **When** an admin/ma_tech operator views a skill
   **Then** `SkillDetail` gains an **Export** action (in the existing action-button cluster) that calls a new `exportSkill` API client and triggers a browser download of the signed bundle via the presigned-URL anchor pattern; and `SkillRegistry` gains an **Import** affordance (in the header, beside "Register Skill") that reuses the `SkillBundleUpload` primitive (presign → PUT → `bundle_key`) then calls a new `importSkill` API client, surfacing the inline 422 (`BUNDLE_SIGNATURE_INVALID` / `BUNDLE_TAMPERED` / `INVALID_EXPORT_BUNDLE`) on failure and navigating to the newly-created skill on success. Icons come from the existing `Icon` component (`download` for export, `upload`/`archive` for import) — never emoji.

## Tasks / Subtasks

> **Scope locked at story creation — do not re-litigate:**
>
> **(L1) Phase 1 only — export/import FILE bundles; in-app service-to-service promote is Story 11.5 (design + stub, NOT built here).** The ADR sequences file-based export/import first precisely because it needs *no* cross-environment networking or identity — it is "serialize → sign → download → upload → verify" entirely within a single environment's existing primitives. Do NOT build any cross-environment call, VPC reachability, or source-env→target-env authentication in this story. ([core-architectural-decisions.md#Environment-Promotion — "Why export/import first"](../../planning-artifacts/architecture/core-architectural-decisions.md))
>
> **(L2) Signing = HMAC-SHA256 with a per-environment key in `Settings`, stdlib `hmac` only.** There is NO existing bundle-signing infrastructure (verified: zero `hmac`/`cryptography` bundle sign/verify in `app/`). HMAC-SHA256 over the canonical envelope bytes, key read from a new `Settings.BUNDLE_SIGNING_KEY` field, verified with `hmac.compare_digest`. This mirrors the platform-boot-credential discipline of `ANTHROPIC_API_KEY`/`SECRET_KEY` (a platform-wide boot credential, injected from Secrets Manager via ECS — NOT the per-skill-execution `SecretsProvider`, which is reserved for connector creds fetched at run time). Do NOT reach for asymmetric signatures (JWT/RSA/Ed25519) — HMAC is the ADR's implied direction, adds zero dependencies, and both environments share the key out-of-band (Secrets Manager). ([secrets.py:1-14](../../../velara-api/app/integrations/secrets.py#L1), [config.py:191-206,256-301](../../../velara-api/app/core/config.py#L191))
>
> **(L3) Import re-creates through the EXISTING immutable registry path — no forked "import register" branch.** For a bundle export, re-assemble the artifact members + `manifest.json` into a ZIP and drive them through the SAME `_create_skill_from_bundle` path (`_process_bundle` re-extracts + re-validates the manifest + entrypoint — the import gets the registration-time contract check for free). For an inline export, drive the inline single-object path. The ONLY import-specific logic is: signature/tamper verification (before extraction) + preserving the source version string + landing `draft` with no cert copy. Story 11.2's `validate_entrypoint_contract` and `parse_code_driven_manifest` re-check the imported artifact exactly as for any registration. ([skill_service.py:522-642](../../../velara-api/app/services/skill_service.py#L522))
>
> **(L4) Import lands `draft`, copies NO certification rows.** The create paths already hardcode `lifecycle_state="draft"` ([skill_service.py:583,755](../../../velara-api/app/services/skill_service.py#L583)). Do NOT insert `certification_records` rows from the envelope — the two-key gate reads the *target org's* rows for the current version and MUST require both keys again ([certification_service.py:410-475](../../../velara-api/app/services/certification_service.py#L410)). The envelope may carry source cert *metadata* for provenance display, but it grants no trust and is never written to `certification_records`.
>
> **(L5) Export must handle BOTH artifact shapes.** A skill version is either single-object inline (`is_bundle=False`, `artifact_key` = one fetchable object) or a multi-file bundle (`is_bundle=True`, `artifact_key` = a prefix, members in `artifact_set`). Export MUST serialize both — do NOT assume every skill is a bundle. Fetch inline bytes via `storage.get(artifact_key)`; fetch each bundle member via `storage.get(f"{artifact_key}/{member['path']}")` and verify its `sha256` on the way out. ([execution_service.py:698-747](../../../velara-api/app/services/execution_service.py#L698))

## Dev Notes

### Why this story is bounded the way it is

11.4 is the **Phase-1 mechanism** for environment promotion (SKL-05). The ADR is explicit that file-based export/import ships *first* — before the Phase-2 in-app service-to-service promote (11.5) — precisely because it needs none of the cross-environment networking, VPC reachability, or source→target identity that promote requires. Export/import is "serialize → sign → download → upload → verify," entirely within one environment's existing single-environment primitives. **The four hard lines from the ADR bind this story:**

1. **Signed + content-addressed bundle.** Manifest + artifact files + version metadata, content-addressed (the existing sha256 discipline) and HMAC-signed with a per-environment key from `Settings` (never in the bundle, never in code). ([core-architectural-decisions.md#Environment-Promotion, decision 1](../../planning-artifacts/architecture/core-architectural-decisions.md))
2. **Import re-creates immutably AND lands non-`client_ready`.** A validated bundle recreates the skill+version through the immutable registry path; a tampered/unsigned bundle is rejected. The imported skill lands `draft` so **target-environment certification must run** before it serves clients there. *Trust does not copy across environments by file transfer* — a dev-certified skill is not thereby prod-certified (Part-11 §11.50/§11.70 bind the signature to a version *in that system*). (ADR decision 2 — **the load-bearing rule**.)
3. **PHI/IP boundary.** The bundle **leaves the platform** (an admin downloads it), so it contains ONLY the skill artifact — code, manifest, schemas, metadata — and NEVER any invocation input/output, document reference, or run history. Export serializes the registry `SkillVersion` + artifact, nothing from `invocation_*`/`audit_*`/ingest. *If it isn't the skill itself, it isn't in the bundle.* (ADR decision 3.)
4. **Promote (11.5) is the same decision minus the file — designed, not built here.** (ADR decision 4.)

### The critical design nuance — signature authenticates the ENVELOPE; per-file re-hash authenticates the FILES

The HMAC signature covers the canonical `velara-export.json` bytes only. The artifact ZIP members live *outside* the signed envelope bytes, so the signature alone does NOT prove the files match. The envelope therefore carries a per-file `sha256` manifest + a content-address digest, and **import must recompute both from the extracted member bytes and compare** — a byte flipped in an artifact member (with the envelope untouched) is a `BUNDLE_TAMPERED`, distinct from a byte flipped in the envelope (which breaks the signature → `BUNDLE_SIGNATURE_INVALID`). Both checks are mandatory. This is exactly the two-layer discipline the registry already uses at run time: `code_driven_executor.py` re-hashes every member against its recorded `sha256` on materialization ([code_driven_executor.py:256-259](../../../velara-api/app/services/code_driven_executor.py#L256)) — the export/import path mirrors it across the environment boundary.

### Canonicalization is load-bearing — sign and verify MUST serialize identically

HMAC is over *bytes*, so the exporter and importer MUST produce the identical canonical envelope bytes: `json.dumps(..., sort_keys=True, separators=(",", ":"))`. A single whitespace or key-order drift makes every signature fail verification even for an untampered bundle. Compute the signature over the envelope with its own `signature` key ABSENT, then attach it; on import, `pop("signature")` before re-canonicalizing. There is a JWT precedent for HS256 in the codebase ([auth.py:334](../../../velara-api/app/integrations/auth.py#L334)) but do NOT route bundle signing through JWT — a bundle envelope is not a token; a bespoke, explicit `hmac.new(...).hexdigest()` over canonical JSON is clearer and adds nothing.

### Import lands `draft` and copies NO certs — the whole point of the story

Both create paths already hardcode `lifecycle_state="draft"` ([skill_service.py:583,755](../../../velara-api/app/services/skill_service.py#L583)); do NOT override it toward the source's state. Do NOT insert any `certification_records` row from the envelope: `assert_certified_for_client_ready` reads the *target org's* rows for the *current version* ([certification_service.py:436-475](../../../velara-api/app/services/certification_service.py#L436)), and it must find zero and require both keys again. The envelope may carry source cert metadata for a "was certified in dev by X on Y" provenance line in the UI, but that string grants no trust and is never written to `certification_records`. If the dev tempts themselves to "carry the cert to save re-signing," STOP — that reverses the load-bearing ADR rule and defeats the BAA gating.

### `create_skill` hardcodes `1.0.0` — import must call `_create_skill_from_bundle` directly to preserve the version

`create_skill` always mints `initial_version="1.0.0"` ([skill_service.py:684](../../../velara-api/app/services/skill_service.py#L684)); an import that preserves the exported `2.3.0` must call `_create_skill_from_bundle` **directly** with `initial_version=envelope["version"]` (that function already parametrizes `initial_version`, [skill_service.py:528](../../../velara-api/app/services/skill_service.py#L528)). For the inline shape there is no equivalent bundle helper — either add a thin `_create_skill_from_inline(..., initial_version=...)` sibling factored out of `create_skill`'s inline branch ([skill_service.py:715-812](../../../velara-api/app/services/skill_service.py#L715)), or extend `create_skill` with an optional `initial_version` param (defaulting to `1.0.0` so every existing caller is unaffected). Dev's call — document it in Completion Notes; the extend-with-optional-param route is smaller and keeps one factory.

### Export must handle BOTH artifact shapes (L5) — do NOT assume every skill is a bundle

A version is single-object inline (`is_bundle=False`, `artifact_key` = one fetchable S3 object, e.g. a prompt/code skill or a manifest-as-artifact code-driven hybrid registered inline) OR a multi-file bundle (`is_bundle=True`, `artifact_key` = a prefix, members in `artifact_set`). The run-time executor already branches on `is_bundle` ([execution_service.py:698-747](../../../velara-api/app/services/execution_service.py#L698)); export mirrors that branch. The round-trip integration test MUST cover both — an inline-only export path that silently drops bundle members (or vice-versa) is the primary regression risk.

### Reuse the EXISTING presign + staged-fetch for import upload (do NOT reinvent)

Import upload rides the SAME `/bundle/presign` → PUT → `bundle_key` flow Story 11.1 built. `fetch_staged_bundle` ([skill_service.py:486-516](../../../velara-api/app/services/skill_service.py#L486)) already enforces (a) the `bundle-staging/{caller_org}/` prefix IDOR guard (the 🔴 CRITICAL fix from the 11.1 review — an arbitrary `bundle_key` could read/delete cross-org objects), and (b) the HEAD size cap before loading into memory. Import MUST route through it — do NOT add a second, less-guarded read path. The FE's existing `SkillBundleUpload` primitive + `presignBundle`/`putBundleToS3` clients are reused verbatim.

### There is NO zip-writer in the backend — the export serializer is net-new

`bundle_extractor.py` only *reads* ZIPs (`extract_bundle`); grep confirms zero `zipfile.ZipFile(..., "w")`/`writestr` in `app/services/`. The export serializer is the **inverse of `extract_bundle`** — a small `zipfile.ZipFile(io.BytesIO(), "w", ZIP_DEFLATED)` writing the envelope member + each artifact member in a deterministic (sorted-path) order. Import re-extracts with the existing `extract_bundle`, inheriting every security guard for free.

### There is NO signing infrastructure — it is net-new, but the ingredients exist

Verified: no `hmac`/`cryptography` bundle sign/verify anywhere in `app/`. But `hashlib.sha256` is the established checksum discipline, `SecretsProvider`/`SecretsManagerProvider` exist ([secrets.py:42-106](../../../velara-api/app/integrations/secrets.py#L42)), `SECRET_KEY`/`ANTHROPIC_API_KEY` show the platform-boot-credential pattern (a `Settings` field + boot-guard, injected from Secrets Manager via ECS — [config.py:191-206,256-301](../../../velara-api/app/core/config.py#L191)), and JWT HS256 shows HMAC in use for tokens ([auth.py:334](../../../velara-api/app/integrations/auth.py#L334)). HMAC-SHA256 with `Settings.BUNDLE_SIGNING_KEY` + stdlib `hmac.compare_digest` is a clean fit — zero new dependency. **`SecretsProvider` is NOT the right seam** for the signing key: it is reserved for per-skill connector credentials fetched at *execution* time ([secrets.py:1-14](../../../velara-api/app/integrations/secrets.py#L1)); a bundle signing key is a platform-wide boot credential, so it belongs in `Settings` + the boot-guard exactly like `ANTHROPIC_API_KEY`.

### Two new audit events (none exist for skill authoring/export)

Only `EVENT_ADMIN_SKILL_ADAPTER_PROPOSED` exists today ([audit.py:58](../../../velara-api/app/models/audit.py#L58)); there is no `_CREATED`/`_EXPORTED`/`_IMPORTED`. ### IP / PHI discipline (house invariants)

- The exported bundle contains ONLY the skill artifact (code, manifest, schemas, metadata) — NEVER invocation input/output, document refs, run history, or audit rows. Export reads only `Skill`/`SkillVersion` + `StorageProvider`; it touches no `invocation_*`/`audit_*`/ingest table.
- Never log artifact bytes, envelope field *values*, the signing key, or S3 keys — only IDs, the content-address digest, model-free counts, version strings, outcome.
- Rejection messages name the failure class (`BUNDLE_TAMPERED`, `INVALID_EXPORT_BUNDLE`, …) and at most the envelope-version identifier — never the offending bytes or field values (mirror the `INVALID_CODE_DRIVEN_MANIFEST` IP-safe error construction, [code_driven_hybrid.py:224-233](../../../velara-api/app/services/code_driven_hybrid.py#L224)).
- Export/import read staged objects only from the caller's own `bundle-staging/{org}/` prefix (via `fetch_staged_bundle`) — no second, less-guarded read path.
