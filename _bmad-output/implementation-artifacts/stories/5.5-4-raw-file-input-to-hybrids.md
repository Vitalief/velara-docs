# Story 5.5.4: Raw-File Input to Code-Driven Hybrids

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a code-driven hybrid skill,
I want the original uploaded file available by path (not only pre-parsed text),
So that I can do my own ingestion — including page rasterization for a vision pass.

## Acceptance Criteria

**AC1 — Raw file materialized to a path and passed as `input_path` (FR-ING-01):**
Given an invocation with `file_ref_ids`
When a code-driven hybrid executes
Then each referenced file's **original uploaded bytes** (the raw `ingest_key` object, NOT the parsed text) are materialized to a file inside the skill's ephemeral venv workspace, and the entrypoint is called with `input_path` = the first materialized file's path (and `input_paths` = the full ordered list — see AC4), augmenting the prior text-only `build_context_input` seam so the skill can read the raw bytes (e.g. for page rasterization).

**AC2 — S3-key-reference + PHI discipline preserved:**
Given the raw-file path is provided
When the skill reads it
Then the S3-key-reference pattern and PHI discipline are preserved — the file is pulled from `StorageProvider` by key (`ingest_storage.get(ref.ingest_key)`), written **only** to the ephemeral venv workspace, cleaned up after the run (the existing `finally: shutil.rmtree(workspace)` already covers the new input subdir), and never logged (log file count + byte size only, never filename content or bytes).

**AC3 — Prompt / LLM-driven-hybrid text seam unchanged:**
Given a prompt or LLM-driven-hybrid skill
When it executes
Then its existing text-only context seam (`build_context_input` → `[Document N]` blocks) is unchanged — raw-file materialization applies **only** to the code-driven path. The `_run_prompt`, `_run_code`, and the LLM-driven branch of `_run_hybrid` are not modified in behavior.

**AC4 — Multi-file contract (locked decision):**
Given an invocation with N `file_ref_ids` (N ≥ 0)
When a code-driven hybrid executes
Then ALL N files are materialized in `file_ref_ids` order; the runner calls the entrypoint with `input_path` = the **first** materialized path (or `None` when N = 0 — back-compat with the 5.5.3 stub) AND `params["input_paths"]` = the ordered list of all materialized paths. The entrypoint signature stays `func(input_path, output_dir, params)` — the multi-file list rides inside `params`, so a single-file skill keeps using `input_path` unchanged.

### Where this story plugs in — the one-line summary

Story 5.5.3 built the code-driven executor with `input_path=None` hardcoded (a deliberate stub — see `code_driven_executor.py:16` docstring and Step 6 at lines 265-270). **This story replaces that stub with real raw-file materialization.** It is the natural, pre-scoped continuation: the 5.5.3 Dev Notes explicitly say "raw file materialization is Story 5.5.4". Almost all the work lands in `code_driven_executor.py`; the rest is plumbing one new storage provider down the call chain and one new `ingest_service` helper.

1. **Multi-file contract:** materialize ALL `file_ref_ids`; `input_path` = first materialized path (back-compat with the 5.5.3 singular runner contract and the single-PDF client skill), `params["input_paths"]` = the full ordered list. The entrypoint signature `func(input_path, output_dir, params)` does NOT change — multi-file rides inside `params`. (Chosen over single-first-only and over reject-if->1.)

### VERIFIED against the real client skill (`velara-protocol-extractor`, deliverable 2026-06-15)

The actual client bundle was read during story creation (`~/Downloads/velara-protocol-extractor/`, `src/velara_extractor/`). The findings below are ground-truth, not assumptions:

**⚠️ TWO BREAKING runner-contract mismatches found — they are NOT 5.5.4's to fix, but 5.5.4 edits the runner so they MUST be recorded here (logged to deferred-work.md for 5.5.3-followup / 5.5.6):**

1. **Entrypoint arg shape — `params` dict vs named kwargs.** The real entrypoint is `velara_extractor.plugin:run(input_path, output_dir=None, *, model=None, consensus_runs=1, enrich=True, emit_excel=True)` — it takes **keyword-only named args**, NOT a `params` dict. The 5.5.3 runner calls `func(input_path=..., output_dir=..., params=json.loads(argv))`, which raises `TypeError: run() got an unexpected keyword argument 'params'` against the real skill — **the client skill would not start.** 5.5.4 keeps the `params`-dict runner contract (its scope is `input_path`/`input_paths`, not the kwarg shape). The fix — the runner must splat invocation inputs as `**params` into the entrypoint (`func(input_path=..., output_dir=..., **params)`), OR 5.5.6 ships a thin platform-adapter shim that maps `params` → named kwargs — is a **5.5.6 / 5.5.3-followup decision**, logged in deferred-work.md. Do NOT silently "fix" it inside 5.5.4 by changing the call convention without that decision (it would change the contract every code-driven skill binds to).

2. **`artifacts` is a dict, not a list.** The real `run()` returns `"artifacts": {"canonical_json": path, "excel": path|None}` (a dict keyed by role). The 5.5.3 `CodeDrivenResultEnvelope` declares `artifacts: list[dict]`. Validating the real envelope today raises `CODE_DRIVEN_ENVELOPE_ERROR`. This is **Story 5.5.5's** job (multi-artifact persistence reshapes the envelope), but it means a real-skill E2E will fail envelope validation until 5.5.5 lands — confirms the 5.5.5 dependency in the epic. The skill also returns an extra top-level `billing_grid` key (pydantic ignores unknown fields by default — non-breaking, but 5.5.5 may want to persist it).

These two are why the headline gate is **5.5.6 after 5.5.1–5.5.5 all done** — 5.5.4 alone cannot run the client skill, and it is not supposed to. 5.5.4's job is strictly to put the raw file on disk at `input_path`; this section exists so the dev agent does not mistake an inherited contract gap for a 5.5.4 regression.

### CRITICAL: raw bytes vs. parsed text — do not confuse the two keys

`FileReference` carries **two** S3 keys (`app/models/file_ref.py`):
- `ingest_key` → the **raw uploaded bytes** in the **ingest bucket** (`S3_INGEST_BUCKET`). ← **THIS story uses this.**
- `parsed_content_key` → the **extracted UTF-8 text** in the **output bucket** (`S3_OUTPUT_BUCKET`). ← the text seam (`build_context_input`) uses this; **NOT** this story.

The existing `build_context_input` (`ingest_service.py:377-406`) reads `parsed_content_key` from `output_storage`. The whole point of 5.5.4 is to give the skill the RAW bytes (a PDF the skill will rasterize), so it must read `ingest_key` from `ingest_storage`. Using the wrong key/bucket here is the most likely silent bug — the skill would get text where it expects a PDF.

### The storage-threading gap (Task 1 detail)

`get_ingest_storage()` exists (`app/integrations/storage.py:207`) but is **not** imported in the worker today — `execution_tasks.py:113` imports only `get_output_storage, get_skill_storage`, and `execute_skill` (`execution_service.py:297`) has no `ingest_storage` param. The three providers are distinct buckets:

| Factory | Bucket setting | Holds |
|---|---|---|
| `get_skill_storage()` | `S3_SKILL_BUCKET` | skill artifacts (manifests/bundles) |
| `get_output_storage()` | `S3_OUTPUT_BUCKET` | parsed text + job outputs (canonical.json) |
| `get_ingest_storage()` | `S3_INGEST_BUCKET` | **raw uploaded files** ← new consumer |

Thread `ingest_storage` worker → `execute_skill` → `_run_hybrid` → `_run_code_driven_hybrid` → `run_code_driven_hybrid`. Do not add it to `_run_prompt`/`_run_code` — they don't need it, and keeping it off those signatures makes AC3 self-evident in the diff.

After this story it must read `input_path=argv[2]`, shifting `output_dir`→`argv[3]`, `params`→`argv[4]`. The `cmd` list in the entrypoint subprocess (lines 302-316) MUST be updated in lockstep. Add a unit test that asserts arg positions so a future edit can't silently desync the runner from the cmd. Use a `__VELARA_NONE__` sentinel for the 0-file case (an empty-string argv slot is ambiguous and shell-fragile).
