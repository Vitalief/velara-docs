# Story 5.5.2: Schema-Versioned Skill Output Contract (Registry Metadata)

Status: done

## Story

As a platform developer building downstream skills,
I want a skill's output schema and `schema_version` to be first-class, queryable registry metadata, and a way for a skill to declare the upstream schema it consumes,
So that a pipeline of skills can bind to one canonical, versioned data contract.

## Acceptance Criteria

**AC1 — Output contract exposed via skill read API:**
Given a code-driven hybrid declaring an `output_schema` + `schema_version`
When it is registered
Then the schema + version are persisted as first-class registry metadata and exposed via the skill read API (FR-REG-03 activated, FR-OUT-05)

**AC2 — Consumes-declaration persisted:**
Given a skill declaring an upstream schema (and version) it consumes
When it is registered
Then the consumes-declaration is persisted, so a downstream skill's input can be validated against the upstream producer's contract before a pipeline runs (FR-OUT-05)

**AC3 — Per-version output contract independently resolvable:**
Given two versions of a skill with different `schema_version`s
When I query the registry
Then each version's output contract is independently resolvable (supports the immutable-version contract — FR-REG-01)

### Critical design: Skill-level vs. Version-level schema_version

**Skill-level `schema_version`** (from 5.5.1): reflects the **current version's** output contract. Updated on each `create_version()` call for code-driven hybrids. Optimized for single-version lookups (list endpoint, quick reads).

**SkillVersion-level `schema_version`** (this story, AC3): a **per-version snapshot** — so `/api/v1/skills/{id}` with its embedded `current_version` can show the version's own schema_version independently. This is the "independently resolvable" guarantee from AC3.

Both columns coexist. They will be equal for the current version. The SkillVersion-level one is the authoritative per-version record.

### Critical pattern: create_skill() flow at version row creation

Read `skill_service.py:214-311` carefully. The SkillVersion row is flushed before `skill.current_version_id` is set. The version_row object is the local Python instance — set `version_row.schema_version` and `version_row.consumes` on it before the flush:

```python
# After `skill.schema_version = manifest.schema_version` (code-driven) or None:
version_row = SkillVersion(
    skill_id=skill.id,
```

### Critical pattern: update_metadata() for consumes PATCH

Check `update_metadata()` in `skill_service.py` — it applies only the fields present in `SkillMetadataUpdate`. The `SkillMetadataUpdate` schema uses `reject_explicit_null()` mode-before validator for non-clearable fields, but `consumes` should be clearable (setting it to null is valid — "no longer consumes anything"). Do NOT add consumes to the `reject_explicit_null` guard. The update pattern should be:

```python
if update.consumes is not _unset:   # or however unset is detected
    skill.consumes = update.consumes  # allows None to clear
```

Check how other nullable-and-clearable fields are handled in `update_metadata`. `output_schema` and `input_schema` are in `SkillMetadataUpdate` and are clearable — mirror that pattern exactly.

### Edge cases to handle correctly

1. **Prompt/code skills with `consumes`:** All `runtime_type` values can declare `consumes`. The `consumes` field is not restricted to code-driven hybrids. A prompt skill that consumes a code-driven hybrid's output should be able to declare that.

2. **LLM-driven hybrid with `consumes`:** Same — valid. The consumes-declaration is purely registry metadata; it has no effect on execution in this story.

3. **Updating `schema_version` on a code-driven skill via PATCH:** `schema_version` is derived from the manifest only — it is NOT included in `SkillMetadataUpdate`. Do not allow `PATCH /api/v1/skills/{id}` to override `schema_version` directly. Only `create_version()` with a new manifest can change it.

4. **`SkillVersionRead.schema_version` for existing rows:** After migration 0014, all existing `skill_versions` rows have `schema_version = NULL`. This is correct — they were created before code-driven hybrid support. No backfill needed.

5. **Circular import guard:** `schemas/skill.py` currently does NOT import from `skill_service.py`. If you need `_SEMVER_RE`, prefer defining a local copy to avoid introducing a circular dependency.

### SkillMetadataUpdate and `reject_explicit_null` guard

Check `app/schemas/skill.py` for the `reject_explicit_null()` validator. It rejects explicit `null` for fields that are required to have values once set. The `consumes` field is different — it starts as `null` and can be explicitly cleared back to `null` via PATCH. Do NOT add `"consumes"` to the `reject_explicit_null` field list. Use the same approach as `input_schema` and `output_schema` in the update schema (they appear in `SkillMetadataUpdate` at lines 278-279 and are nullable).
