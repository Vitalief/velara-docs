# Story 5.5.3: Code-Driven Hybrid Executor (Phase-1 Venv)

Status: done

## Story

As a consultant,
I want the platform to execute a code-driven hybrid skill in an isolated per-skill venv with its dependencies installed, network egress permitted, platform secrets injected, and resource limits enforced,
So that a skill that drives its own multi-pass/vision LLM orchestration runs on the platform.

## Acceptance Criteria

**AC1 — Sub-dispatch inside existing hybrid branch:**
Given a registered code-driven hybrid
When it is invoked and the Celery `run_skill` task dispatches it
Then `execution_service.execute_skill` enters the existing `runtime_type == "hybrid"` branch and **sub-dispatches by manifest shape** to the new code-driven executor (the LLM-driven `_run_hybrid` path is unchanged for `{system, tools[], code}` manifests) (FR-EXE-03)

**AC2 — Per-skill venv with deps, egress, and secrets:**
Given the executor runs the bundle
When it starts
Then a per-skill venv is created with the lockfile deps installed; **network egress to the manifest's declared destinations is allowed** (in contrast to the adversarial sandbox's hard socket block); `ANTHROPIC_API_KEY` and the skill's declared `requires` secrets are injected as env via `SecretsProvider` (reusing the `_resolve_credentials` seam); and wall-clock + memory + output limits are enforced (FR-EXE-05)

**AC3 — Entrypoint called; result envelope captured:**
Given the executor runs the bundle
When the skill's entrypoint `module:callable` is called with `{input_path, output_dir, params}`
Then the platform captures the result envelope `{status, schema_version, canonical, artifacts[], qa}` returned by the entrypoint (this envelope is handed to Story 5.5.5 for full multi-artifact persistence; for now return a single canonical-JSON artifact key)

**AC4 — Isolation + resource limits; timeout → `failed`, never silent hang:**
Given a buggy or runaway skill
When it runs
Then its venv isolation + resource/timeout limits prevent it from affecting other skills or platform stability; a timeout or crash yields `failed` (via the existing `EXECUTION_TIMEOUT` / `SKILL_EXECUTION_ERROR` error codes), never a silent hang

## Dev Notes

### Architecture: Where the code-driven executor fits

`execution_service.py` is the **runtime router** (module docstring: "Routes on `skill.runtime_type`"). Its `_run_hybrid()` function is the entry point for ALL `hybrid` skills. Story 5.5.3 adds detection at the TOP of `_run_hybrid`:

```
execute_skill()
  └─ runtime_type == "hybrid"  →  _run_hybrid()
        ├─ is_code_driven_manifest(artifact_raw) == True
        │   └─ _run_code_driven_hybrid()  ← NEW (5.5.3)
        │       └─ code_driven_executor.run_code_driven_hybrid()  ← NEW
        └─ is_code_driven_manifest() == False  (LLM-driven — Story 3.5)
            └─ parse_hybrid_artifact() + _agentic_loop()  [UNCHANGED]
```

This keeps the routing clean: `_run_hybrid` detects shape; the actual execution is in `code_driven_executor.py`.

### Current `_run_hybrid` flow (PRESERVE this exactly for LLM-driven):

1. Calls `_resolve_credentials()` → `extra_env`
2. Fetches artifact bytes → `parse_hybrid_artifact()` (Story 3.5 — LLM-driven)
3. Assembles user content from location block + file_ref_ids + free inputs
4. Runs `_agentic_loop()` in threadpool (ToolServer sandbox, manual tool-use loop)
5. Calls `_persist_output()` → `(output_key, result_metadata)`

**The NEW code-driven path intercepts at step 2** — after fetching artifact bytes, BEFORE calling `parse_hybrid_artifact`. If `is_code_driven_manifest(artifact_raw)` returns True, redirect to the code-driven executor and return immediately (do NOT fall through to the LLM-driven path).

### `_resolve_credentials` seam — REUSE exactly

Location: `execution_service.py:240-292`. This function:
- Takes `skill.requires` tokens → fetches from `SecretsProvider` → returns `{ENV_VAR: value}` dict
- Already handles: reserved-key blocking, collision detection, logging count only (no values)
- **Raises `MissingSecretError`** on miss → `_map_error_code` already maps to `MISSING_CREDENTIAL`

For code-driven hybrids, call it identically. Then ALSO inject `ANTHROPIC_API_KEY` separately (it is not in `skill.requires` — it's a platform-provided key, not a user-declared credential):

```python
creds = _resolve_credentials(skill=skill, secrets_provider=secrets_provider)
api_key = secrets_provider.get_secret("ANTHROPIC_API_KEY")  # raises MissingSecretError if not set
injected_env = {**os.environ, **creds, "ANTHROPIC_API_KEY": api_key}
```

`ANTHROPIC_API_KEY` IS in `_RESERVED_CREDENTIAL_KEYS` (blocked from `requires`), which is correct — it should come from the platform, not from the skill's own `requires` tokens.

### Bundle artifact shape

The code-driven hybrid artifact stored in S3 is the manifest JSON, NOT a zip file. The zip IS the artifact content that gets packaged when a real skill is registered — but what the platform stores and fetches from S3 is `initial_content` (the manifest JSON string). The zip extraction is a FUTURE concern once raw file delivery is in place.

**Reconsideration for Story 5.5.3:** The manifest declares `requirements` as a lockfile text string inline — NOT a zip file. There is NO actual zip bundle at this stage. What `skill_storage.get(artifact_key)` returns is the manifest JSON bytes. The "bundle" is a conceptual description in the epic. For Phase 1 (5.5.3), the platform has:
- The manifest JSON (including `requirements` text, `entrypoint`, `output_schema`, `schema_version`)
- The entrypoint module path

The actual skill code module (`velara_extractor.plugin`) must be present in the Python environment. **For Phase 1 (local dev testing), the skill code is pre-installed** or installed via the `requirements` lockfile which should include the skill package itself as an editable install. The platform is NOT a zip-extraction platform yet — that requires the raw-file-by-reference work (Story 5.5.4).

**Revised AC3 implementation for 5.5.3:** The entrypoint invocation uses the venv (with deps from `requirements`) and calls `manifest.entrypoint`. If the skill code is a package installable via `requirements` (e.g., `velara-extractor @ file:///path/to/local-pkg`), it works. For unit tests, use a trivially-installable module (e.g., create a temp module file directly in the workspace).

**Simplified runner for this story:** Write a tiny `runner.py` script to the workspace, run it with the venv Python, capture stdout as JSON. The key architectural seam is the subprocess boundary + env injection + timeout.

### Timeout sandwich — CRITICAL invariant

From the execution engine patterns doc:
```
entrypoint wall-clock limit:  1200s  (EXECUTION_TIMEOUT_S — the subprocess timeout)
Celery soft_time_limit:       1260s  (SoftTimeLimitExceeded → EXECUTION_TIMEOUT)
Celery hard time_limit:       1320s
```
This order MUST be preserved. The venv creation + pip install timeouts (120s + 300s) are additive overhead BEFORE the entrypoint starts — they eat into the Celery budget but not into `EXECUTION_TIMEOUT_S`. If pip install takes >300s, the task fails with `CODE_DRIVEN_EXECUTION_ERROR` (pip timeout), not `EXECUTION_TIMEOUT`.

### `asyncio.run()` constraint

`run_skill` is a sync Celery task. It calls `asyncio.run(_execute())`. Inside `_execute()`, `execute_skill()` is called as `await`. The code-driven executor's `run_code_driven_hybrid()` is called as `await` inside `_run_hybrid()` which is already in the threadpool via `run_in_threadpool`. The subprocess calls (`venv create`, `pip install`, `runner.py`) are synchronous — they run inside `run_in_threadpool` (which executes in a thread, not the event loop). Use `run_in_threadpool` to wrap all synchronous subprocess/filesystem calls inside the async executor function. This is the same pattern used in `_run_code` and `_run_hybrid`.

### PHI / IP discipline

- **Never log** `requirements` text, skill code, input content, or output content verbatim
- **Log:** package count (e.g. `packages_installed=7`), entrypoint name (metadata, not PHI), workspace path (ephemeral), job_id, duration_ms
- **Cleanup:** `shutil.rmtree(workspace, ignore_errors=True)` in `finally` — no workspace on disk after the run
- **Output key:** `jobs/{job_id}/canonical.json` — same key pattern as existing output; never expose the key to the skill (injection vector)
- **`ANTHROPIC_API_KEY`:** injected into env, never logged, never included in result_metadata
