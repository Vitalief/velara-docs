## ⭐ Headline Finding — Why the auth step missed so much (Project Lead's question)

**The question:** *"The auth step missed so many necessary functionalities — why was that?"*

**Root cause — the epic modeled the admin's action, not the invited user's journey.**
Epic 10 was scoped as **provisioning**: *create the identity, invite it, grant it access* (10.1 → 10.2 → 10.3). What was never written down as first-class scope was the **login lifecycle the provisioned user actually walks through**. The plan carried an implicit, wrong assumption: *"10.1 creates the Cognito user, Story 7.3 already does login, therefore login works."* It didn't — **10.1 provisioning and 7.3 Cognito login were never connected end-to-end.**

Consequences, all auth-lifecycle gaps that had to be retro-fitted or are still open:

- **First-login password challenge (10.4)** — a freshly-provisioned user is in Cognito `FORCE_CHANGE_PASSWORD`. `login()` never inspected `signIn()`'s `nextStep`, so a *correct* temp-password login rendered the misleading blanket **"Invalid username or password."** **Not a regression — a scope gap** left by 10.1/7.3 never being wired together.
- **Forgot / reset password (10.5)** — **completely absent**: no route, no "Forgot password?" link, no `resetPassword`/`confirmResetPassword` call anywhere in the FE. User-reported the same session.
- **Still NOT covered even after 10.4/10.5** (explicitly deferred, not built): other `signIn` nextSteps — **MFA, `RESET_PASSWORD`, `CONFIRM_SIGN_UP`** — still fall through to the blanket string; `completeNewPassword` ignores its own returned nextStep (chained challenges). Deferred until MFA/reset handling is genuinely needed (MFA is OFF in dev today).

**How the gaps were found: a manual smoke test, not planning.** On 2026-07-03 an `ma_tech` created a consultant user and *tried to log in as them* — that single act surfaced both 10.4 and 10.5. If no human had attempted a real provisioned-user login, both would still be missing.

**Systemic lesson (Winston):** two different actors were in play — the **admin** who provisions, and the **invited user** who activates and later recovers an account. We storyboarded only the first. The invited-user auth lifecycle (temp password → first login → set password → forgot-password-later → MFA/other challenges) should have been enumerated as its own thread at epic-planning time.

**Action:** **AG1 (below)** — when an epic provisions an identity/credential, storyboard the *end user's* full lifecycle as first-class stories, not an afterthought of the admin flow.

## What Went Well

1. **Forward-planned reuse actually worked.** 10.2 built the `AddUserOverlay` Stepper to take a `steps` array and *deliberately omitted* the engagement selector from its create form "because it feeds 10.3's grant step." 10.3 then extended 2→3 steps by changing one array literal — **no Stepper change**. Cleanest forward-planned reuse in the project so far.
2. **The `AuthProvider` seam paid compounding dividends.** Every BE addition extended the Protocol (`create_user` → `resend_invite`) rather than bypassing it; `AuthPrincipal` from 10.1 made 10.3's handoff a one-liner (`created.user_id → create_grant`); 10.4's same-card `phase` sub-state pattern was reused wholesale by 10.5.
3. **Reviews caught inverted assumptions, not just bugs.** 10.3's **consultant silent-grant** is the standout: the team *assumed* the backend `INTERNAL_ROLE_NOT_GRANTABLE` guard would 422 a consultant-into-client grant. Empirical verification proved it **silently SUCCEEDED** (the guard keys on the grant *body role*, not the opaque `user_id`) — creating an ADR-forbidden client-role grant for an internal user. Worse than expected, caught before merge. Similarly 10.1's review caught that the **invite email never sent at all**.
4. **Security discipline held across an identity epic.** `role: Literal["client","consultant"]` makes admin/ma_tech structurally un-mintable from the UI; `org_id` is server-side from the caller (with a review-added test proving a smuggled body `org_id` is ignored); 10.4's review closed an **open public `SignUp` API on an invite-only pool**; 10.5 is enumeration-safe by construction (pool `prevent_user_existence_errors=ENABLED` + FE swallow) and its review closed a theoretical enumeration oracle.
5. **Honest verification notes improved over the epic.** 10.2's review caught the Dev Agent Record overstating that Playwright "confirmed" flows that had actually 405'd on a stale container; 10.3/10.4/10.5 then wrote scrupulously honest "manual E2E NOT performed / left to operator" notes.

## What Was Hard (cross-story patterns)

1. **Auth-lifecycle scope gap** — the headline (above). Planning modeled the admin, not the invited user; two of five stories were mid-epic discoveries.
2. **Cognito quirk class — defaults that silently do the wrong thing.** The `DesiredDeliveryMediums` default-is-SMS miss (10.1), the email-as-username → `{username}`-renders-a-UUID greeting bug (10.4), `UserStatus` being a top-level key not an attribute (10.2), `confirmationCode`-not-`code` (10.5). Every story after 10.1 front-loaded a "Cognito facts you must not re-derive" Dev Notes block precisely because of this.
3. **Test-infra tax (carried from Epic 8/9, still recurring).** Dev-shim module-global state leaks across tests (`_SEED_USERS`/`_SEED_EMAILS` under `@lru_cache`, no reset fixture — 10.1; `_INVITED_EMAILS` bare set — 10.2, deferred); image-bakes-source → stale container → **405 on create/resend, browser verification abandoned** (10.2); `email-validator` rejecting `.test` TLDs → `@*.example` in bodies (10.1→10.2).
4. **FE state-hygiene / stale-state bugs** — same class in 10.2 (`setTimeout` no unmount cleanup), 10.3 (`grantError` lifecycle, Done-node snapshotted-not-re-resolved, null-email "()"), 10.5 (`resetSuccess`/`resetEmail` never reset). React state not cleared/snapshotted at the right lifecycle boundary.
5. **Live verification could not be done in-loop** — the standing "don't reconfigure the live Cognito stack" rule (correct) means the epic accrued a live-verification backlog rather than confirming as it went.

## Action Items

### Team agreements (no ticket)
- **AG1 — Storyboard the end user's full lifecycle, not just the admin's action.** When an epic provisions an identity or credential, enumerate the *invited/affected user's* complete journey as first-class scope: create → activate (first-login challenge) → recover (forgot/reset) → other challenges (MFA, confirm-signup, chained). Epic 10 modeled only the admin who provisions; the user's login lifecycle was found by manual smoke test, costing two retro-fitted stories (10.4/10.5) with several branches still deferred. *Extends Epic 9's AG1 story-creation checklist with an actor-journey lens.*
- **AG2 — Manual smoke test IS the safety net; make it a planned step, not an accident.** Both mid-epic discoveries came from one human trying a real provisioned-user login. For any auth/identity work, schedule a "log in as the thing you just created" smoke pass **during** the epic, on a disposable stack — don't let it be the thing that happens to reveal a whole missing story.
- **AG3 — Keep the 3-layer adversarial review with empirical service-model verification.** It caught the invite-never-sends, the phantom `AdminResendInvitation` API, and the consultant silent-grant — all things static reading missed. Non-negotiable for identity/security work.
