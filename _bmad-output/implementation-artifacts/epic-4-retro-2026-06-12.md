## What Went Well

### The 4-1 ltree Foundation Held
Native `ltree` column with GiST indexes, org-segment sanitization for any JWT org_id, and a get-or-create-org race fixed with a SAVEPOINT. This load-bearing schema supported three UI stories stacked on top of it (4-2, 4-3, 4-4) with **no structural rework**.

### Single-Screen-Grown-In-Place
4-2 → 4-3 → 4-4 were not three rewrites. The `EngagementsScreen` was grown from 2 levels to 4 (4-3), then re-architected from selection-state to URL routing (4-4) — and stayed coherent. Shared `ChildListCard`/`DetailActions` kept it DRY as it grew.

### Adversarial Review Caught a Real Data-Integrity Bug
The 4-3 AC6 cascade bug is a model case: a study selected via **search or the Project-detail card** (never expanded in the tree) had an empty `locationsCache` → the delete dialog showed no "N locations" warning, the cascade deleted nothing, the API then 409'd. AC6's core promise was silently defeated. **Both the Blind Hunter and Edge Case Hunter caught it independently.** Fixed by fetching the live location list at confirm time. Then in 4-4, review caught the follow-on — the same fetch respected the global 30s `staleTime`, making the "live list" comment false — and added `staleTime: 0`. The system worked exactly as intended: a cache-population-by-UI-side-effect bug that would have corrupted data in production was caught pre-deployment.

### Retro-Document Habit Held
Epic 3 retro action item 3 ("save retro documents going forward") was honored — Epic 3's retro exists on disk, and this document continues the habit.

### Non-Atomic Client-Side Cascade Delete (Data-Integrity Risk)
The 4-3/4-4 study-delete cascade deletes child locations in a client-side loop, then the study. A mid-loop non-404 failure (403/500) leaves locations permanently deleted, the study intact, and the user in a half-deleted state. Correctly deferred (the real fix is a backend cascade endpoint), but it is a **data-integrity** risk, not cosmetic.

**Decision (with Project Lead):** Add a small **backend cascade-delete endpoint** (`DELETE ...?cascade=true`). Retires both this risk and the 4-1 child-guard friction.

## Key Insights

1. **Grow one screen in place, don't rewrite.** The single-`EngagementsScreen` discipline (shared cards, additive store writes, preserved behaviors across stories) is why a selection-state → URL-routing refactor in 4-4 didn't shatter 4-2/4-3. Keep this pattern for multi-story UI surfaces.
2. **Adversarial review earns its cost on data-integrity bugs.** The 4-3 cache-by-side-effect cascade bug + its 4-4 staleTime follow-on are exactly the class of bug that ships silently and corrupts data weeks later. Two independent layers caught it. Keep 3-layer review on every story.
3. **Deferred cross-epic threads need a hard checkpoint, not a log line.** Story 3-7 is the second epic where a deferred item nearly drifted. The fix that worked here: make it a *named prerequisite with an owner and a position in the sequence*, not a backlog entry.
4. **Action items slip when the current epic doesn't force them.** Epic 3 items 1 & 2 slipped because Epic 4 had no execution work. Tie each carried action to a concrete trigger in the *next* epic (item 1 → Story 3-7; item 2 → Epic 5 kickoff) so it can't float.
