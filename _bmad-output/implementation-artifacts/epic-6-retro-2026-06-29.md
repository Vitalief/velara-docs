## What Went Well

1. **DB-level immutability landed correctly.** Migration 0015's append-only triggers mirror the audit-log pattern (0006). This directly honored Winston's 5.5-retro flag: "enforce immutability at the DB layer, not just the app." Intent → artifact.
2. **The adversarial review caught a hollow test.** 6.1's original immutability test received a 405 from route resolution without ever touching the database — the migration-0015 triggers (the actual Part 11 backstop) had zero coverage. The review caught it; a real DB-level UPDATE/DELETE-raises test was added.
3. **Clean auto-advance UX (6.3).** The methodological key auto-advances the skill to `client_ready` server-side, inside `record_certification` after the cert row commits. One action moves the lifecycle.
4. **Live-infra integration discipline held.** Carried from the 5.5 retro — the DB-level immutability assertions only mean something against real Postgres triggers; mocks would have sailed through.

---

## What We Learned (Key Insights)

1. **"Everything went well" is when patterns hide.** The production code across Epic 6 was solid. The recurring weakness was **test integrity** — four of 6.1's five review patches were tests that passed without proving anything:
   - immutability test = a 405 from routing, never touching the DB
   - `certified_at` test passed even if the server had accepted a client-supplied value
   - eligibility unit tests mocked `select` away, so the security-relevant org/version `.where()` filters were never exercised
   
   **The adversarial review is the control that catches this.** The team's explicit decision: do NOT add a new "no mock-masked tests" rule — the review already covers it, and adding process for a working control is overhead. The one non-negotiable that makes the review real: keep running compliance/security-critical tests against live infrastructure.

2. **Mock-heavy tests are where false confidence lives.** Mocking `select` or asserting on route-level status codes feels like coverage but proves nothing about the security/compliance behavior. Real assertions run the real query / hit the real trigger.

3. **A process item that survives two retros without producing an artifact should be killed or re-formatted — not auto-renewed.** (Winston's "two-retro rule," adopted this retro.)
