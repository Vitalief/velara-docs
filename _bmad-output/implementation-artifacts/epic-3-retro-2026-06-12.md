## What Went Well

### Architectural Seams Held
The 3-1 scaffold (job lifecycle, terminal-state guard, audit append-only) was solid enough that all three subsequent runtimes (3-3, 3-4, 3-5) extended it without structural changes. The `execute_skill` router pattern plugged in cleanly.

### Timeout Sandwich Pattern (Story 3-4)
The "sandbox (1200s) < Celery soft (1260s) < Celery hard (1320s)" ordering was a non-obvious invariant that was established correctly on the first try in 3-4 and inherited cleanly by 3-5.

### Code Review Catching Real Production Risks
74 patches across 7 stories fixed genuine pre-production risks: stranded jobs on broker failure, forged control signals via subprocess pipe, unbounded stdout memory reads, XML injection in PDF rendering, dispatch orphans on broker failure. The adversarial review structure worked.

### Post-Review Live Testing Loop (Story 3-6)
Flat Markdown rendering in PDF output was not caught by automated tests or code review — it was discovered via live ARROW-protocol testing after review passed. The team turned it around same day by introducing `markdown-it-py`. The Project Lead was able to view and improve actual output PDFs — end-to-end visibility was achieved within Epic 3.

### Connector Framework Absorbed (Story 3-8)
Added mid-epic via correct-course resequencing, delivered cleanly without disrupting in-flight stories.

## Challenges & Growth Areas

### Same Patterns Rediscovered Across Stories
Three patterns were re-encountered independently across multiple stories:
1. **asyncio.run() / event-loop conflict in Celery tasks** — hit in 3-1, 3-2, 3-3, 3-4 (resolution: service-level testing, not task-level)
2. **Dispatch atomicity** (broker failure after DB write → orphaned job) — patched independently in 3-2 and 3-3
3. **Terminal-state guard** — established in 3-1, re-confirmed/re-patched in 3-2 and 3-3

Root cause: lessons lived in individual story dev notes but weren't propagated forward. No living "Execution Engine Patterns" document existed.

### Epic Scope Expectation Gap
The Project Lead expected to be able to run skills through the UI after Epic 3. The actual boundary — Epic 3 delivers backend execution infrastructure, Epic 5 (Run Console) delivers the UI — was not visibly communicated. The epic title "Skill Execution Engine" is technically accurate but implies more end-to-end visibility than was actually delivered.

### No Epic 2 Retrospective Document
Epic 2's retrospective is marked `done` in sprint-status but no document was saved to disk. Institutional memory from Epic 2 is lost. This is the first retro with a saved document.

### Story 3-7 Is an Unfinished Thread
Location-dependent fan-out is a named Epic 3 story sitting in `backlog`. It cannot be created until after Epic 4 Story 4-1 (hierarchy ltree CRUD). This needs an explicit checkpoint to avoid being forgotten.

## Key Insights

1. **Real usage catches what tests and review cannot.** The Markdown rendering gap in 3-6 only surfaced under live invocation with real LLM output. Build real testing into the workflow, not just suite + review.
2. **Pattern documentation pays off immediately.** Rediscovering the same asyncio/dispatch patterns 3-4 times across a single epic is avoidable. A living patterns doc would have paid for itself in Story 3-2.
3. **Deferred items compound.** 18 logged deferral items is manageable, but several (hierarchy_path validation, S3Connector.fetch NotImplementedError) are load-bearing for Epic 4 and Epic 5. They need explicit callouts in story creation, not just log entries.
4. **Epic milestone framing matters for project-level confidence.** "What a consultant can do after this epic" is different from "what was built." Both matter — only the second was communicated.
