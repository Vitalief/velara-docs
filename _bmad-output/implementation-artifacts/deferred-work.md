## Production bug fixed post-9.3 (user-reported, 2026-07-02) — internal-caller invocations always attributed to "org"

- **RESOLVED — `queue_invocation`'s non-LD branch hardcoded `hierarchy_path="org"` for EVERY unrestricted (internal) caller, ignoring any project/study context.** `app/api/v1/invocations.py`'s non-location-dependent branch had `if scope.unrestricted: hierarchy_path_str = "org"` with no check of `body.study_id` at all — this was the deliberate, documented behavior from Story 8.4 D1 ("Internal (`unrestricted`) callers keep the existing `"org"` behavior unchanged"). In practice this meant every internal user's run of a non-LD skill from the Engagements screen or Run Console — regardless of the Study/Project they were visibly scoped into — silently landed at the org root, breaking audit attribution (the exact symptom that surfaced via the 9.3 Audit Log UI: a real invocation showed `hierarchy_path="org"` instead of the study/project it was run from). Compounding it: the FE (`RunConsole.tsx`, both context-first and skill-first modes) never sent `study_id`/`project_id` in the request payload for non-LD skills in the first place — the `showLocationSelector`-gated payload construction only populated location fields for location-dependent skills, so even fixing the backend alone would not have helped until the FE started sending the context.
  **Fix (2026-07-02):** (1) `InvocationRequest` schema gained `project_id`/`client_id` alongside the existing `study_id` (no prior field could express "run at Project/Client level" — only Study). (2) `queue_invocation`'s non-LD branch now resolves `body.study_id`/`body.project_id`/`body.client_id` (most-specific first) via `hierarchy_service.get_study`/`get_project`/`get_client` for BOTH unrestricted and scoped callers — falling back to `"org"` only when no context is supplied at all (e.g. invoked from the Skill Registry with nothing selected). (3) `RunConsole.tsx`'s two `handleRun()`s (context-first + skill-first) now send the resolved `study_id`/`project_id` for non-LD skills instead of `{}`. Regression tests added: `test_non_location_skill_internal_caller_attributes_to_study`, `..._to_project`, `..._no_context_still_defaults_to_org` (BE); `RunConsole.test.tsx` payload assertions rewritten to expect the real context instead of `{}`. This also resolves (supersedes) the "D1 non-LD client-invocation widening merged UNDER A WAIVER" item from the Epic 8 retrospective below — the underlying 8.1 org-global boundary question is now moot for BOTH scoped and unrestricted callers, since context resolution is symmetric across both.
