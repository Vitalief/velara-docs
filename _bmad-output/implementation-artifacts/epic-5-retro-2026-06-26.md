## What Went Well

### 202-Async Model Held Across All 7 Stories
The async dispatch pattern, atomic commit-then-dispatch, and 17-code error map (user-safe messages, never raw exception text) shipped in 5-1 and was never broken by the 6 stories built on top of it. Zero invocation-contract regressions. The `extra="forbid"` guard held — no unknown field ever reached the worker.

### Adversarial Review Caught Real Bugs Every Story
- 5-2: forbidden color tokens (brand-drift trap), typecheck was silently RED (3 TS6133)
- 5-3: canRun allowed retired/no-version skill through to 422; LD skill + 0 locations was a silent dead-end
- 5-4: child (fan-out) jobs appeared as top-level History rows; mutations left stale activeJobId in store; fan-out polling never stopped on persistent error
- 5-7: skill mutations (create/update/transition) didn't invalidate the new paged cache — stale rows for ~30s after any write

All of these would have shipped to production without the 3-layer review.

### Full-Stack Scope Decision on 5-4 Was Correct
Identifying upfront that Jobs History needed a net-new backend list endpoint — and locking the scope decision before implementation — prevented a false-start. The PageMeta pattern established here was directly reused in 5-6.

### Pagination Tiebreaker Caught and Backfilled
5-6 review caught the missing tiebreaker on `created_at desc` (non-unique → unstable pagination). The fix was applied to both `list_skills` and `list_jobs`, preventing a pre-existing pagination bug in 5-4's endpoint from going unnoticed.

## What Went Wrong — The Honest Breakdown

### Failure 1: Consumer Audit Not Done Before 5-6 Broke listSkills

When Story 5-6 changed the `listSkills` response shape from `Skill[]` to `{items, page}`, the story plan named "one FE consumer" (5-7). There were **three**: `SkillRegistry`, `useProjectSkills` (→ Run Console), and `AppBar` (→ ⌘K whole-catalog palette).

All three broke the moment 5-6 shipped. Story 5-7 had to fix the regression first — unwrapping `.items` and keeping `Skill[]` for the three consumers — before adding any net-new pagination behavior. This is recoverable rework, but it was avoidable.

**Root cause:** No pre-story consumer audit. The story assumed a single consumer because it was the most obvious one. A 10-minute grep before writing the story would have surfaced all three.

**Impact:** ⌘K palette and Run Console skill picker were broken between 5-6 and 5-7 shipping. The primary discoverability surface for skills was degraded.

### Failure 2: Skills Registry Search is Permanently Current-Page-Only

The skills registry table search box filters only the loaded page — not the full catalog. The backend has no `?q=` free-text search parameter (`skills.py:88-95` confirmed — only `page/per_page/status/tag`).

This was known before 5-6 was written. It was logged in `deferred-work.md` as a HIGH follow-up. It was not escalated to the PM as a product decision. It was not added to the backlog as a story.

The result: pagination shipped without search. For any catalog with more than 50 skills (one page), the registry table search is misleading — users expect it to find any skill, and it only finds skills on the current page. The `deferred-work.md` note and the ⌘K hint in the UI are honest documentation of a limitation, but they don't fix the product gap.

**Root cause:** `deferred-work.md` is a dev notes file, not a product backlog. HIGH items need PM review and a story or an explicit "accept this limitation" decision. Neither happened.

**Impact:** Search experience is degraded and will stay that way until backend `?q=` + frontend wiring ships. Needs two stories.

### Failure 3: Epic 5.5 Rework — We Built the Wrong Hybrid Runtime

The `velara-protocol-extractor` — the client's first real skill, the headline deliverable of the entire platform — does not fit any execution shape we built in Epic 3. Not prompt, not code, not the LLM-driven hybrid (platform drives, skill provides sandboxed stdlib tools).

The skill is code-driven: Python code orchestrates 3 LLM passes via `client.messages.stream()` directly. It requires external dependencies (`pdfplumber`, `PyMuPDF`, `openpyxl`), network egress to Anthropic, a raw PDF input, and produces a schema-versioned canonical JSON + Excel workbook with a QA `status: ok|blocked` gate.

Our hybrid runtime inverts the driver relationship. Our sandbox blocks network. Our executor assumes stdlib-only tools. None of this can host the client's skill.

Epic 5.5 is 6 stories of new platform capability — code-driven executor, schema-versioned output contract, raw file input, venv isolation, multi-artifact output, blocked job state — that would have been 1–2 stories if the client skill's execution shape had been known before Epic 3 was designed.

**Root cause:** We designed the hybrid runtime against an abstract "hybrid" concept, not the client's actual deliverable. The client skill existed (or its shape was knowable) before Epic 3. We didn't ask for it or analyze it before writing the execution design.

**Impact:** 6-story epic of rework. The platform works correctly for what we built, but not for the client's actual use case. This is the most expensive miss of Epic 5.

## Post-Retro: Client Skill Analysis (velara-protocol-extractor)

As part of closing this retrospective, the actual client skill at `/Users/apple/Downloads/velara-protocol-extractor` was analyzed to verify the Epic 5.5 plan before the epic starts.

### Execution Shape (confirmed)
- Code drives; LLM is callee via `client.messages.stream()` (Anthropic SDK ≥0.69)
- 3 sequential LLM passes: (A) study design, (B) SoA matrix (with optional multi-run consensus), (C) context enrichment
- Entrypoint: `velara_extractor.plugin:run(input_path, output_dir, model, consensus_runs, enrich, emit_excel)`
- Fully synchronous, single-threaded; no async/await; platform can queue via Celery without modification

### Dependencies (venv required)
- `pdfplumber ≥0.11` — PDF text/table extraction
- `PyMuPDF ≥1.24` — page rasterization (base64 PNG for vision pass); MUST be in venv (silent no-op if missing is dangerous)
- `openpyxl ≥3.1` — Excel workbook output
- `anthropic ≥0.69`, `pydantic ≥2.6` — likely pre-available but must be pinned

### Input Contract
- `input_path`: absolute filesystem path to PDF (born-digital or scanned)
- Config overrides via env: `ANTHROPIC_API_KEY` (required), `VELARA_MODEL`, `VELARA_TEXT_BUDGET`
- DOCX declared in manifest but **not implemented** — must be documented as known limitation

### Output Contract
```json
{
  "status": "ok" | "blocked",
  "schema_version": "0.3.0",
  "canonical": { ProtocolExtraction },
  "billing_grid": [ { row } ],
  "artifacts": {
    "canonical_json": "/path/to/stem.canonical.json",
    "excel": "/path/to/stem.procedures.xlsx" | null
  },
  "qa": { "releasable": bool, "egregious": [...], "warnings": [...] }
}
```
- `status: blocked` = egregious QA flags (no_activities, no_instances) — **first-class job outcome, not a failure**
- Two artifact files written to local filesystem

### Epic 5.5 Plan Assessment: No Re-plan Needed
The 5.5 stories (5.5-1 through 5.5-6) match the skill's actual requirements. The plan is correct.

**One architectural decision needed before 5.5-3:** Platform workspace temp-dir pattern — executor pre-mounts a local temp dir, copies input PDF in, skill writes artifacts to it, platform picks them up via StorageProvider. This avoids rewriting the skill's file I/O.

**One AC addition needed for 5.5-6:** Regression corpus run (20–30 real protocols) to verify enum resilience fix (`_LenientStrEnum._missing_()`) — fix exists in code, marked unverified in SAVEPOINT.

**Known limitations to document (not fix in 5.5):**
- No DOCX support (stub only)
- No body-text procedure sweep / FR-15 (v2 feature)
- `consensus_runs > 1` multiplies Anthropic API cost linearly — must be surfaced in UI before users set it
- SAP/prose-only protocols correctly refuse to fabricate but give no guidance — UX follow-up

## Key Insights

1. **Consumer audit is mandatory before any breaking API shape change.** Before writing any story that changes a shared response contract, list every caller (BE + FE). A grep takes 10 minutes and prevents a regression. `listSkills` had three consumers; the story named one.

2. **deferred-work.md is not a product decision.** Logging a HIGH item in a dev notes file is not the same as the PM deciding it's acceptable to ship a degraded experience. HIGH items must be reviewed by PM at the start of each epic: backlog story, in-scope, or explicitly accepted limitation.

3. **Get the client's actual artifact before designing the runtime.** The hybrid executor was designed against an abstract concept. The client's skill inverted the driver relationship and required external deps and network egress. Both were knowable before Epic 3. The cost of not asking was a 6-story rework epic.

4. **Pagination without server-side search is a half-feature.** Shipping pagination to a table that previously had full-catalog client-side search — without a `?q=` backend endpoint — degrades search for any catalog larger than one page. These must be planned together.

5. **`blocked` is not `failed`.** The client skill has a first-class QA gate (`status: blocked`) that means "ran successfully, output didn't meet quality threshold." Platform must surface this as a distinct job outcome, not a disguised failure.
