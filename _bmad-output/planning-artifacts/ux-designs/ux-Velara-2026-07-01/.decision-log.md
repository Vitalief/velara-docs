# Decision Log — Velara Admin UX (Access Control assignment + Client Provisioning)

Run started 2026-07-01. Mode: Create. Facilitator: Sally (UX).

Scope: two internal-admin (consultant/ma_tech, RequireInternal) surfaces from the 2026-07-01 correct-course:
- **Story 8.6** — skill attach/detach interaction on the existing Access Control screen.
- **Epic 10** — client user provisioning (create+invite, user list, create→grant handoff).

## Anchors (facts, not yet decisions)
- Existing prototype `design/internal3.jsx` `AccessControl` (defined :245) already renders the Access Control screen: client tabs → project cards → project-level skill chips + nested study skill chips, each with an "Attach skill" / "Attach" button that is currently a dead stub (`onClick={()=>{}}`).
- Data model `design/data.js` `skillAssignments` (:294) = `{ id, entityType:'project'|'study', entityId, skillId, grantedBy, grantedAt }`, accessor `skillsAt(type,id)`. This matches Winston's ADR `skill_attachment(skill_id, node_id, node_type, org_id, attached_by_user_id, attached_at)` 1:1.
- Winston ADR 2026-07-01: availability = attachment ∩ grant ∩ client_facing/client_ready. Provisioning via `AuthProvider.create_user` → Cognito `AdminCreateUser` default-invite (Cognito emails temp password); `org_id`/`role` set server-side.
- Brand: V3 tokens (`design/styles_v3.css`) — teal `#128F8B`, navy `#323843`, slate `#4C5270`, pink `#F652A0`; Poppins headings / Open Sans body.
- No user-provisioning design exists anywhere (true blank canvas).

## Decisions

**User steer (2026-07-01):** "Do whatever you want, stick to the theme + existing design — EXCEPT the client list at the top of Access Control is not scalable; find a better design for it." → Sally drives everything else to match `internal3.jsx` + V3; the one net-new design problem is the client selector.

**D1 — Scalable client selector (the flagged problem).** Replace the horizontal wrapping button row (`internal3.jsx:261-271`) with a **searchable combobox**: collapsed = a "current client" context trigger (avatar + name + code chip + ▾); open = a search input (filter by name or code) over a scrollable client list (avatar, name, code · lead sub-line; paused clients dimmed but selectable; checkmark on the active one). One client active at a time (screen is already single-client). Reuses the theme's existing `side-search` idiom (`internal.jsx:30-32`) so it feels native. Scales to hundreds of clients. Mock: `.working/admin-surfaces-mock.html` screen 1.

**D2 — Keep the existing assignment layout (8.6).** The project-card → project-level-skills band → nested study rows layout from `AccessControl` (`internal3.jsx:273-353`) is kept verbatim in spirit. Skill chips gain a **detach affordance** (hover reveals an ✕, tinted danger-pink on hover); the previously-dead "Attach skill" / "Attach" buttons now open the attach panel (D3). Mock screen 2.

**D3 — Attach-skill panel (fills the dead stub).** A centered modal opening from any Attach button. Header states the target ("to ONC-204 · Study · <project>"). A search input filters **client-ready skills only**; each row = icon + name + one-line description + Attach button. Already-attached skills appear disabled with an "Attached ✓" badge (prevents duplicates — matches the UNIQUE constraint in Winston's `skill_attachment` ADR). Footer note reinforces the guardrail. **Guardrail by construction:** only `client_facing` + `client_ready` skills are listed, so an internal-only skill *cannot* be attached to a client — the feared mistake is designed out, not merely warned. Mock screen 3.

**D4 — Provisioning as one guided flow, two homes (Epic 10).** Create+invite is a **3-step guided flow** (1 Create & invite → 2 Grant access → 3 Done), NOT three disconnected chores. Step 1: a small form (name / email / client-engagement / role) with a callout explaining Cognito sends the secure temp-password invite (Velara never stores the password — aligns with Winston's provisioning ADR). Step 2 **reuses the existing Access Control grant control**, pre-filled to the new user (the create→grant handoff, Story 10.3) — no context switch, no re-typing. Mock screens 4-5.

**D5 — "Users" surface = its own nav tab (CONFIRMED 2026-07-01).** A top-level **Users** nav tab (peer to Skill Registry / Access Control), scoped by the same client selector (D1). List columns: user (avatar+name), email, engagement access summary, status (Invited / Active pill), row action (Resend invite for invited). "Add user" launches the create+invite flow (D4).

**D6 — Attach = slide-in RIGHT side panel (CONFIRMED, supersedes D3's centered modal).** Docks on the right; the project/study cards stay visible so chips appear live as the admin attaches across several studies without reopening. Same contents as D3 (target header, client-ready-only search, disabled already-attached rows, guardrail footer). Optimizes for the real rhythm — setting up a whole engagement in one sitting.

**D7 — Detach = confirm dialog (CONFIRMED, supersedes D2's one-click).** Hover ✕ on a chip → "Detach <skill> from <study>?" → [Cancel] [Detach]. Deliberate friction because detaching removes a client's access to a deliverable. (Contrast: attaching is fluid/bulk via the side panel; removing is a considered beat.)

**D8 — Cross-tab create→grant handoff (resolves the seam D5 created).** With Users as its own tab, the 3-step onboarding flow spans two tabs (create in Users → grant in Access Control). To keep it feeling like ONE motion, the create+invite flow (D4) is a **focused overlay/route launched from Users**, and on successful create it advances IN PLACE to Step 2 "Grant access" (the reused, pre-filled Access Control grant control rendered inside the same overlay — NOT a redirect that dumps the admin into the Access Control tab to re-find Susan). Step 3 "Done" offers "Add another" or "View Susan in Users." So: Users owns the *entry point + the list*; the grant *control* is reused inside the flow; the admin never has to navigate tabs mid-onboarding. The standalone Access Control tab still grants access to existing users independently.

**UX personality (emergent from the 3 picks):** make bulk/constructive work fluid (side panel, live chips), make destructive/creative actions deliberate (detach confirm, guided create). Consistent, human.

**Guardrail theme across both surfaces:** server-set role/org (no self-elevation), client-ready-only attach list, invite-not-password. All internal-admin only (consultant/ma_tech, RequireInternal).
