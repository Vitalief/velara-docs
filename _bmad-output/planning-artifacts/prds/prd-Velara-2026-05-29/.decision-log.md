## Decisions

### D-01 — PRD audience is MA Technologies dev team
- **Decision:** PRD will be used internally by the MA Technologies development team as the primary implementation reference, not handed back to Vitalief as a deliverable.
- **Source:** User statement.

### D-02 — Working mode: Fast path
- **Decision:** Fast path selected. Gaps batched into one consolidated question set; PRD drafted with `[ASSUMPTION]` tags where inferred.
- **Source:** User preference.

### D-03 — Platform hierarchy extended beyond BRD
- **Decision:** The full hierarchy is Organizations → Clients → Projects → Studies (optional) → Skills → Locations. This was not explicit in the BRD. Studies are optional at the Project level; some Projects attach skills directly without a Study.
- **Source:** User statement.

### D-04 — Locations are physical clinical sites within Studies
- **Decision:** Locations represent physical clinical trial sites (hospitals, research centers) and exist only within Studies. Locations are not their own access control boundary — access control remains at the Study/Project level.
- **Source:** User clarification.

### D-05 — Location-dependent skill behavior
- **Decision:** Skills can declare themselves location-dependent. When invoked without a location:
  (a) Platform prompts the user to select a location, OR
  (b) Platform fans out parallel invocations — one per Location in the Study — and returns aggregated results.
  The skill definition (not the caller) declares which behavior applies.
- **Source:** User statement.

### D-06 — Phase 1 UI is minimal
- **Decision:** Phase 1 includes a minimal web interface for skill management, certification workflow, and basic invocation. Full client-facing web portal is Phase 2 (BRD INV-03).
- **Assumption confirmed by:** User did not object.

### D-07 — Claude invocation via platform API intermediary
- **Decision:** INV-01 is implemented as a proxy pattern. Claude calls the platform endpoint; the platform resolves and executes the skill server-side; only output is returned. Skill internals never leave the server.
- **Source:** Assumed (user confirmed the BRD `????` notes were their own open questions, not blockers). Assumption A8.

### D-08 — Single organization in Phase 1
- **Decision:** Phase 1 deploys Vitalief as the only Organization. Data model supports multi-org for future portability. Assumption A1.
- **Source:** BRD + assumption, not contradicted by user.

### D-09 — BRD open `????` items deferred
- **Decision:** The `????` annotations in the BRD (INV-01, ING-01, OUT-01, EXE-01) were added by the user and are their own open questions. Treated as deferred; not blocking PRD completion.
- **Source:** User statement.

### D-10 — Core priority: skill registry + document-in / document-out execution
- **Decision:** The most important Phase 1 capability is the skill registry combined with file ingest → skill execution → structured output. All other capabilities support this core loop.
- **Source:** User statement.

### D-12 — Epic 5.5 FR amendments applied (2026-06-18)
- **Decision:** Applied 8 FR edits to §5 (both `prd/5-functional-requirements.md` and `Velara-PRD-full.md`, kept in sync) per the approved `sprint-change-proposal-2026-06-18.md` (Epic 5.5 — Code-Driven Hybrid Skills & Canonical Output Contract).
- **Trigger:** Client delivered the first real foundation skill (`velara-protocol-extractor`) — a **code-driven hybrid** (its code drives, the LLM is a callee; multi-file; third-party deps; needs network egress + raw-PDF for a vision pass; emits a schema-versioned canonical JSON + Excel + a QA envelope with `ok|blocked`). It fits none of prompt/code/hybrid as built. **No 4th runtime** — the hybrid runtime is widened with a second execution shape.
- **Edits:**
  1. **REG-01** (amend) — the immutable versioned artifact MAY be a multi-file bundle (zip/dir).
  2. **REG-03** (amend) — input/output schema are load-bearing + schema-versioned for code-driven hybrids.
  3. **EXE-03** (amend) — hybrid supports two shapes: (a) LLM-driven (existing), (b) code-driven trusted (code drives, calls the LLM, bears deps, egress + injected secrets + raw-file; certification-gated).
  4. **EXE-05** (amend) — sandboxing is tiered: adversarial sandbox for code/LLM-driven-hybrid; isolated-but-trusted for code-driven hybrids (stability not hostility; trust via certification).
  5. **EXE-09** (new, P1) — `blocked` job state on a QA egregious gate; blocked outputs not consumable downstream. *(One FR covering state + non-consumption, per user.)*
  6. **ING-01** (amend) — skills may receive the raw uploaded file by reference/path, not only pre-parsed text.
  7. **OUT-01** (amend) — outputs include canonical JSON conforming to a versioned schema; a run may produce multiple mixed-type artifacts.
  8. **OUT-05** (new, P1) — schema-versioned output contract: platform persists output schema + `schema_version`; downstream skills declare the upstream schema they consume.
- **Decisions taken this session (user):** amend REG-01 directly (not fold the bundle note into REG-03); keep EXE-09 as one FR (state + downstream non-consumption together).
